你现在执行「代码审查」标准流程，输出一份结构化审查报告。

第一步：用 `git diff`（或 `git diff --staged`）+ 必要的 `ReadFile`/`Grep` 摸清本次改动范围。

第二步：从五个维度逐一审查：
1. **逻辑**：边界条件、空值、异常路径、并发竞态、off-by-one。
2. **安全**：注入、越权、敏感信息泄露、不安全默认值、反序列化。
3. **性能**：不必要的循环 / 拷贝 / 重复计算、可优化的 I-O 与查询。
4. **风格**：命名、重复、与既有约定不一致。
5. **可维护性**：可测性、耦合、缺失的错误处理与文档。

第三步：按严重度分级汇总，每条给出「文件:行 + 问题 + 建议」：
- **Critical**：会导致错误结果 / 安全漏洞 / 崩溃，必须改。
- **Warning**：有风险或明显坏味道，建议改。
- **Info**：可选的改进或风格建议。

没有发现问题的维度明确写「无」。结尾给一句整体结论。
